## Behavioral Contracts

**BC-1: closed_loop in Reasoning section**
GIVEN a user reads the "Reasoning (Multi-Turn with Context Accumulation)" subsection
WHEN they scan the key-fields list
THEN they can find `closed_loop`, its default (true for multi-turn clients), and the opt-out semantics (`false` = open-loop, all rounds pre-stamped)

**BC-2: context_growth behavioral explanation**
GIVEN a user reads the `context_growth` field description
WHEN they choose between `"accumulate"` and omitting the field
THEN they understand: `"accumulate"` causes linear input growth per round (each round's input = all prior inputs + all prior outputs + new user turn), creating growing KV cache pressure and enabling prefix-affinity routing benefits; the omitted/empty case uses freshly sampled tokens per round with no cross-round prefix sharing

**BC-3: open-loop vs closed-loop conceptual prose**
GIVEN a user reads the Multi-Turn Conversations section
WHEN they first encounter the two modes
THEN a prose paragraph explains the distinction: closed-loop (default) means each round waits for the prior reply and uses actual completion time to schedule the next round; open-loop (`closed_loop: false`) means all rounds pre-stamped using a `1 µs/output-token` heuristic

**BC-4: arrival process semantics for multi-turn**
GIVEN a user reads the Arrival Processes section (or the multi-turn section)
WHEN they configure a multi-turn reasoning client
THEN they understand that arrival process governs session-start inter-arrival times (not individual request times), think_time_us governs inter-round spacing, and single_session:true fires the arrival process exactly once per client

**BC-5: YAML round-trip test for closed_loop**
GIVEN a WorkloadSpec YAML string containing `closed_loop: false`
WHEN parsed through yaml.Unmarshal + GenerateWorkload
THEN the client is treated as open-loop (wl.Sessions is empty, requests have RoundIndex > 0)
AND if the YAML tag on ClientSpec.ClosedLoop were removed, this test would fail

### Task 2: Expand context_growth description

**File:** `docs/guide/workloads.md` — `context_growth` bullet (~line 440)

Replace:
```
- `multi_turn.context_growth`: `"accumulate"` to prepend all prior input+output as context, or omit for independent rounds
```

With:
```
- `multi_turn.context_growth`: controls how each round's input is constructed.
  - `"accumulate"`: round N's input = all prior rounds' inputs + all prior rounds' outputs + freshly sampled new user turn. Input length grows linearly with round index, creating expanding KV cache pressure and enabling prefix-affinity routing reuse across rounds — the right mode for modeling realistic chat or reasoning sessions.
  - `""` (omit): each round uses only freshly sampled tokens. Input length is stationary across rounds; no cross-round prefix sharing. Use this for agent workloads that do not maintain conversation history.
```

### Task 3: Add conceptual open-loop vs closed-loop prose

**File:** `docs/guide/workloads.md` — after the multi-turn YAML example (~line 188), before "## CLI Distribution Mode"

Add a short prose paragraph explaining:
- Default (closed-loop): each round's arrival time is set by the simulator after the prior round completes — `next_arrival = completion_time + think_time_us`. This means actual server latency propagates into inter-round spacing; under high load the session stretches.
- Open-loop (`closed_loop: false`): all rounds are pre-stamped before simulation. Uses `1 µs/output-token` as a heuristic for expected completion time. Server latency does not affect inter-round spacing — useful for reproducing pre-defined workload traces.

**Verify:** Both modes described in the multi-turn section before CLI section.

### Task 4: Add arrival process semantics note for multi-turn

**File:** `docs/guide/workloads.md` — "Arrival Processes" section (~line 253) or in the Multi-Turn Conversations section

Add a callout or note explaining:
- For non-multi-turn clients: arrival process governs inter-arrival time between individual requests.
- For multi-turn clients with `single_session: false` (default): arrival process governs inter-session start times — one new session begins per IAT tick, producing `max_rounds` rounds per session.
- For multi-turn clients with `single_session: true`: arrival process fires exactly once per client, determining when that client's single session starts.
- Within any session, inter-round spacing is always governed by `think_time_us`, not the arrival process.

Also add the "traffic source vs user" distinction with the concurrent-users example from the issue comment.

**Verify:** Arrival process semantics documented for multi-turn case.

### Task 5: Add YAML round-trip test for closed_loop

**File:** `sim/workload/generator_test.go`

Add a new test function `TestGenerateWorkload_ClosedLoop_YAML_RoundTrip` that:
1. Defines a YAML string containing `closed_loop: false` for a reasoning multi-turn client
2. Calls `yaml.Unmarshal` (using `gopkg.in/yaml.v3`) to parse into `WorkloadSpec`
3. Calls `GenerateWorkload` on the result
4. Asserts `wl.Sessions` is empty (open-loop path taken, not closed-loop)
5. Asserts at least one request has `RoundIndex > 0` (all rounds pre-generated)

This test fails if the YAML struct tag on `ClientSpec.ClosedLoop` is removed or mistyped.

Note: `gopkg.in/yaml.v3` does NOT have `yaml.UnmarshalStrict` (that was yaml.v2). Use the decoder
pattern with `KnownFields(true)` per R10. This is the same pattern as `cohort_test.go:342-344`
and `spec.go:209-211`.

```go
func TestGenerateWorkload_ClosedLoop_YAML_RoundTrip(t *testing.T) {
    // This test verifies YAML parsing path for closed_loop: false.
    // Existing tests use Go struct literals; this test specifically catches
    // regressions to the yaml:"closed_loop,omitempty" struct tag on ClientSpec.ClosedLoop.
    const yamlStr = `
version: "2"
seed: 42
category: language
aggregate_rate: 10.0
clients:
  - id: reasoning-yaml
    tenant_id: t1
    slo_class: standard
    rate_fraction: 1.0
    arrival:
      process: poisson
    input_distribution:
      type: constant
      params:
        value: 50
    output_distribution:
      type: constant
      params:
        value: 20
    reasoning:
      reason_ratio_distribution:
        type: constant
        params:
          value: 0
      multi_turn:
        max_rounds: 3
        think_time_us: 1000
    closed_loop: false
`
    var spec WorkloadSpec
    dec := yaml.NewDecoder(strings.NewReader(yamlStr))
    dec.KnownFields(true)
    if err := dec.Decode(&spec); err != nil {
        t.Fatalf("yaml decode: %v", err)
    }
    wl, err := GenerateWorkload(&spec, 10_000_000, 50)
    if err != nil {
        t.Fatal(err)
    }
    // YAML-parsed closed_loop: false must route to open-loop path (no session blueprints)
    if len(wl.Sessions) != 0 {
        t.Errorf("YAML closed_loop:false: expected 0 session blueprints, got %d", len(wl.Sessions))
    }
    // Open-loop path pre-generates all rounds; at least one request must have RoundIndex > 0
    hasLaterRound := false
    for _, req := range wl.Requests {
        if req.RoundIndex > 0 {
            hasLaterRound = true
            break
        }
    }
    if !hasLaterRound {
        t.Error("YAML closed_loop:false: expected RoundIndex > 0 (all rounds pre-generated)")
    }
}
```

The `strings` package import is required (or `bytes` + `bytes.NewReader` — either is fine).

**Verify:** `go test ./sim/workload/... -run TestGenerateWorkload_ClosedLoop_YAML_RoundTrip -v` passes.
