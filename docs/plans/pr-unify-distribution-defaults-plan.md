# Unify Distribution Synthesis Defaults Across `run` and `observe` — Implementation Plan

**Goal:** Make `blis run` and `blis observe` produce identical workload distributions when called with identical distribution flags, by sharing a single set of default constants.

**The problem today:**
`blis run` and `blis observe` register the same eight distribution-shaping flags but with different hardcoded defaults in separate files for six of them. `blis run --rate 10` and `blis observe --rate 10` silently generate different workload shapes with no flags given, making the `observe → calibrate` comparison misleading without explicit flags. The six divergent flags are: `--prompt-tokens-stdev`, `--prompt-tokens-min`, `--prompt-tokens-max`, `--output-tokens-stdev`, `--output-tokens-min`, `--output-tokens-max`.

**What this PR adds:**
1. A single Go `const` block in `cmd/root.go` that defines eight distribution synthesis defaults (six divergent ones + the two means that already agree — all sharing one definition site).
2. Both `runCmd` and `observeCmd` flag registrations reference these constants instead of hardcoded literals.
3. An updated comment in `observe_cmd.go` that accurately describes the new parity.
4. Updated help text for `observe`'s `--num-requests` flag documenting the intentional asymmetry with `run` (0 = spec-driven, not unlimited).

**Why this matters:**
The `observe → replay → calibrate` pipeline measures how accurately the simulator models a real server. If the workloads generated by `observe` and `run` differ in shape by default, the calibration result conflates model error with input distribution skew. Shared defaults eliminate that confound.

**Architecture:**
All changes are confined to `package cmd`. Both `cmd/root.go` (defines constants + `runCmd`) and `cmd/observe_cmd.go` (`observeCmd`) live in the same package, so constants defined in `root.go` are directly accessible from `observe_cmd.go` with no import. No new files, interfaces, or types are introduced.

**Source:** GitHub issue [#863](https://github.com/inference-sim/inference-sim/issues/863), parent tracking issue [#869](https://github.com/inference-sim/inference-sim/issues/869).

**Closes:** Fixes #863.

**Behavioral Contracts:** See Part 1, Section B.

**Construction site audit:** This PR does not add fields to any struct. The flag variables in `observe_cmd.go` and `root.go` are plain `int` package-level variables. No struct construction sites are affected.

**Which flags are included in the shared constants:**
- Included: `--prompt-tokens` (mean, 512), `--output-tokens` (mean, 512), and the six divergent shape flags (stdev/min/max for both). The means already agree; including them ensures the const block covers all distribution synthesis parameters and prevents future drift on mean values.
- Excluded: `--prefix-tokens` (0 on both). Zero is a semantically distinct value meaning "no shared prefix" — it is a feature toggle, not a distribution shape parameter. See Deviation Log.

## Part 1: Design Validation

### A) Executive Summary

This PR fixes a silent default divergence between `blis run` and `blis observe`. Both commands register six distribution flags with the same names but different hardcoded defaults in separate files. The fix is to define one `const` block in `cmd/root.go` and reference it from both flag registration sites.

The `--num-requests` asymmetry (100 on `run`, 0 on `observe`) is intentional and preserved: `run` benefits from a safe non-zero default so quick simulations work out of the box; `observe` correctly requires an explicit bound because hitting a real server with unbounded requests is dangerous. This PR makes the intent explicit via help text and a comment update.

No deviation from issue #863's proposed fix. No dead code. Four files changed: `cmd/root.go` (add const block, update eight flag registrations), `cmd/observe_cmd.go` (update eight flag registrations + one comment + one help string), `cmd/observe_cmd_test.go` (add two test functions), `cmd/root_test.go` (add one test function).

### B) Behavioral Contracts

```
BC-1: Shared distribution shape defaults
- GIVEN the observeCmd and runCmd flags are registered at startup
- WHEN a user calls either command without any distribution flags
- THEN --prompt-tokens, --prompt-tokens-stdev, --prompt-tokens-min,
       --prompt-tokens-max, --output-tokens, --output-tokens-stdev,
       --output-tokens-min, --output-tokens-max
       all have identical default values on both commands
- MECHANISM: both flag registrations reference the same package-level constants

BC-2: Single source of truth for default values
- GIVEN the eight distribution synthesis constants are defined once in cmd/root.go
- WHEN the code is compiled
- THEN no old literal value for the six formerly-divergent defaults appears in
       observe_cmd.go's distribution flag IntVar calls
- MECHANISM: verified by both a value-equality test (runtime) and a source-level
       scan test (static) that checks for the old literal integers

BC-3: Documented num-requests asymmetry
- GIVEN --num-requests has different defaults on run (100) and observe (0)
- WHEN a user reads `blis observe --help`
- THEN the --num-requests help text explains that 0 means "from spec or unlimited"
       and does NOT imply the same default as run
```

### E) Review Guide

The tricky part is verifying BC-2 (single source of truth). The plan uses two complementary tests:
1. A **value-equality test** (runtime): checks that each flag's default value equals the constant. This catches value drift when someone changes one file but not the other.
2. A **source-level scan test** (static): reads `observe_cmd.go` and checks that the old literal integers (`50`, `1`, `2048`) are not present in the distribution flag IntVar calls. This catches literal re-introduction even when the value coincidentally matches the constant.

Reviewers should scrutinize both tests: the value-equality test verifies BC-1; the source-scan verifies BC-2.

The `--num-requests` decision (keep asymmetry, document it) is the most debatable design choice. Reviewers should verify the help text is clear enough that a new user would understand why the defaults differ.

Everything else is mechanical: eight constant references replacing eight literal integers.

```go
// TestRunCmdDistributionDefaults_UseSharedConstants verifies that runCmd's eight distribution
// flag defaults equal the package-level constants (BC-1, BC-2: single source of truth).
//
// What this test catches: if someone changes a constant value, both commands change
// together and the test still passes. If someone bypasses the constants with a different
// hardcoded literal, the test fails.
func TestRunCmdDistributionDefaults_UseSharedConstants(t *testing.T) {
	tests := []struct {
		flag string
		want int
	}{
		{"prompt-tokens", defaultPromptMean},
		{"prompt-tokens-stdev", defaultPromptStdev},
		{"prompt-tokens-min", defaultPromptMin},
		{"prompt-tokens-max", defaultPromptMax},
		{"output-tokens", defaultOutputMean},
		{"output-tokens-stdev", defaultOutputStdev},
		{"output-tokens-min", defaultOutputMin},
		{"output-tokens-max", defaultOutputMax},
	}
	for _, tt := range tests {
		f := runCmd.Flags().Lookup(tt.flag)
		if f == nil {
			t.Fatalf("flag --%s not found on runCmd", tt.flag)
		}
		got, err := strconv.Atoi(f.DefValue)
		if err != nil {
			t.Fatalf("--%s DefValue %q is not an int: %v", tt.flag, f.DefValue, err)
		}
		if got != tt.want {
			t.Errorf("--%s default: got %d, want %d", tt.flag, got, tt.want)
		}
	}
}
```

```go
// TestObserveDistributionDefaults_MatchRunDefaults verifies that observeCmd's eight
// distribution flag defaults are identical to runCmd's defaults (BC-1).
//
// Why this test matters: the observe→calibrate loop compares simulator output against
// real-server output. If the two commands generate different workload shapes by default,
// calibration conflates model error with distribution skew.
//
// If someone changes a constant in root.go, both commands change together and this
// test still passes. If someone accidentally re-introduces a hardcoded literal with a
// DIFFERENT value, this test fails.
func TestObserveDistributionDefaults_MatchRunDefaults(t *testing.T) {
	tests := []struct {
		flag    string
		wantInt int
	}{
		{"prompt-tokens", defaultPromptMean},
		{"prompt-tokens-stdev", defaultPromptStdev},
		{"prompt-tokens-min", defaultPromptMin},
		{"prompt-tokens-max", defaultPromptMax},
		{"output-tokens", defaultOutputMean},
		{"output-tokens-stdev", defaultOutputStdev},
		{"output-tokens-min", defaultOutputMin},
		{"output-tokens-max", defaultOutputMax},
	}
	for _, tt := range tests {
		f := observeCmd.Flags().Lookup(tt.flag)
		if f == nil {
			t.Fatalf("flag --%s not found on observeCmd", tt.flag)
		}
		got, err := strconv.Atoi(f.DefValue)
		if err != nil {
			t.Fatalf("--%s DefValue %q is not an int: %v", tt.flag, f.DefValue, err)
		}
		if got != tt.wantInt {
			t.Errorf("--%s default: got %d, want %d (must match runCmd default)",
				tt.flag, got, tt.wantInt)
		}
	}

	// BC-3: num-requests intentional asymmetry — observe must default to 0, not 100.
	f := observeCmd.Flags().Lookup("num-requests")
	if f == nil {
		t.Fatal("flag --num-requests not found on observeCmd")
	}
	if f.DefValue != "0" {
		t.Errorf("--num-requests default: got %q, want \"0\" (intentional asymmetry with run's 100)", f.DefValue)
	}
}

// literal defaults (50, 1, 2048) appear in observe_cmd.go's distribution IntVar calls
// (BC-2: single source of truth).

// This is a source-level scan test, following the pattern in simconfig_shared_test.go.
// It catches the scenario where someone re-introduces a literal that coincidentally has
// the same value as a constant — which the value-equality test above would miss.
func TestObserveDistributionDefaults_NoHardcodedLiterals(t *testing.T) {
	data, err := os.ReadFile("observe_cmd.go")
	if err != nil {
		t.Fatalf("cannot read observe_cmd.go: %v", err)
	}
	content := string(data)

	// These patterns are the old hardcoded literals that must no longer appear as
	// the default argument in distribution flag IntVar calls.

	forbidden := []string{

		`"output-tokens-max", 2048`,

			t.Errorf("old hardcoded literal found in observe_cmd.go: %q\n"+

		}
	}
}
```
