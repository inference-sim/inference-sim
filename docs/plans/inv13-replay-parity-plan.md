# INV-13 Run/Replay Parity Implementation Plan

**Goal:** Wire PD disaggregation into `blis replay` and replace silent degradation with fail-fast fatals for unsupported features (autoscaler, node pools).

**The problem today:** `blis replay` silently drops PD disaggregation flags (`--prefill-instances`, `--decode-instances`, `--pd-decider`, etc.) and converts a disaggregated run into a homogeneous cluster replay — producing wrong results with no error. Autoscaler and node-pool config are warned on but silently ignored, also producing divergent results.

**What this PR adds:**
1. PD disaggregation fully wired in `replay.go`: all 11 PD fields (prefill/decode instances, decider, transfer params, contention, per-pool scorers, per-pool hardware overrides) are validated and passed to `DeploymentConfig`.
2. Fail-fast for unsupported features: `logrus.Fatalf` instead of warning when autoscaler or node-pool config is present in replay mode.
3. INV-13 formalized: run/replay parity encoded as a named invariant in `invariants.md` and `CLAUDE.md`.
4. Test coverage: PD parity test (library level) + fatal-exit canary tests (CLI level).

**Why this matters:** `blis replay` is a key fidelity tool for calibrating the simulator against real server traces. Silent divergence defeats the purpose of calibration and produces misleading comparisons.

**Architecture:** All changes are in `cmd/replay.go` (CLI layer). PD validation and config construction mirror the existing `runCmd` code in `cmd/root.go` (lines 1379–1492). New tests live in `cmd/replay_test.go`. INV-13 is documented in `docs/contributing/standards/invariants.md` and `CLAUDE.md`.

**Source:** https://github.com/inference-sim/inference-sim/issues/1268

**Closes:** Fixes #1268

**Behavioral Contracts:** See Part 1, Section B.

## Part 1: Design Validation

### A) Executive Summary

This PR fixes a fidelity gap in `blis replay`. When a trace is generated by a PD-disaggregated cluster and replayed, replay silently uses a homogeneous cluster (all PD fields zero-default), producing per-request TTFTs and E2Es that cannot match the original run. The fix mirrors the existing `runCmd` PD validation and config-wiring block (~100 lines in `root.go:1379–1492`) into `replay.go`.

For features genuinely unsupported in replay (autoscaler, node pools), the existing soft warnings are replaced by `logrus.Fatalf`. This converts silent divergence into an explicit, actionable error.

No new interfaces, types, or packages are introduced. The PR is a targeted config-wiring fix with tests.

Deviation from source: the issue mentions `InstanceLifecycle` as also missing, but it is not called out in Track B (fail-fast). This plan does not add fail-fast for `InstanceLifecycle` — it would require identifying a config field that maps to CLI flag exposure, which doesn't exist for `InstanceLifecycle`. Filed as a follow-up note in Deviation Log.

### B) Behavioral Contracts

**BC-1: PD disaggregation wired end-to-end in replay**
- GIVEN a trace exported from a PD-disaggregated `blis run` (2 prefill + 4 decode instances, `--pd-decider always`)
- WHEN `blis replay` is invoked with the same `--prefill-instances 2 --decode-instances 4 --pd-decider always` flags
- THEN the replay simulation uses a disaggregated cluster topology (prefill pool, decode pool); per-request TTFT/E2E distributions match what a fresh run on the same requests produces
- MECHANISM: `replay.go` passes all 11 PD fields to `DeploymentConfig`, same as `runCmd`

**BC-2: Autoscaler config causes fatal exit**
- GIVEN a policy bundle YAML with `autoscaler.interval_us > 0`
- WHEN `blis replay --policy-config bundle.yaml` is invoked
- THEN the process exits with a non-zero exit code and a message mentioning "autoscaler is not supported in replay mode"

**BC-3: CLI `--model-autoscaler-interval-us` causes fatal exit**
- GIVEN `--model-autoscaler-interval-us 500000` is passed to `blis replay` (flag is registered via `registerSimConfigFlags` which replayCmd's `init()` calls — `cmd/root.go:941`)
- WHEN `blis replay` starts up
- THEN the process exits with a non-zero exit code before running the simulation

**BC-4: Node-pool config causes fatal exit**
- GIVEN a policy bundle YAML with `node_pools` containing at least one pool
- WHEN `blis replay --policy-config bundle.yaml` is invoked
- THEN the process exits with a non-zero exit code and a message mentioning "node_pools is not supported in replay mode"

**BC-5: PD validation parity**
- GIVEN `--prefill-instances 4 --decode-instances 0` (decode pool empty, prefill non-zero)
- WHEN `blis replay` starts up
- THEN `ValidatePoolTopology` is called and produces a fatal error (same as `runCmd`)

**BC-6: INV-13 documented**
- GIVEN the invariants.md file
- WHEN a contributor reads it
- THEN INV-13 run/replay parity is present with statement, verification, and evidence sections

### H) Test Strategy

| Contract | Test | Level | Rationale |
|----------|------|-------|-----------|
| BC-1: PD parity | `TestINV13_RunReplayParity_PD` | Library (cluster) | Exercises the full PD simulation path with exact metric comparison; avoids CLI complexity |
| BC-2: Autoscaler bundle fatal | `TestReplayCmd_AutoscalerBundleFatal` | CLI (subprocess) | logrus.Fatalf calls os.Exit; subprocess pattern is the only reliable unit-test approach |
| BC-4: Node pools bundle fatal | `TestReplayCmd_NodePoolsBundleFatal` | CLI (subprocess) | Same rationale as BC-2 |
| BC-5: PD validation | `TestReplayCmd_PD_BasicSmoke` | CLI | Exercises PD flags → ValidatePoolTopology path |
| BC-6: INV-13 documented | Manual read | — | Documentation only |

### I) Risk Analysis

**Risk 1: Subprocess tests are slow or flaky**
- Mitigation: subprocess tests re-run the test binary (fast); each spawns one process with minimal setup. Mark with `t.Parallel()` if needed.

**Risk 2: PD parity test may expose non-determinism in trace export/import**
- Mitigation: the test uses a fixed seed and deterministic token sequences; existing `TestReplayCmd_TraceOutput_Determinism` covers byte-identical export. If parity fails, it will expose a real INV-13 violation.

**Risk 3: Per-pool scorer parsing in replay may fail if scorer names differ from runCmd**
- Mitigation: `sim.ParseScorerConfigs` is the same function called by runCmd; no risk of divergence. When `prefillScorerCfgs`/`decodeScorerCfgs` are nil (flags not set), the cluster simulator falls back to global `RoutingScorerConfigs` — behavior is handled in `cluster.NewClusterSimulator`, not in replay.go.
